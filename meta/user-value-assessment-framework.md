# 仓库用户价值评估框架

> 基于方舟25期众测方法提炼的通用分析框架，已剥离具体测试场景、主体与过程数据。

---

## 一、评估方法

### 定位差异分析

两个仓库从用户视角出发的定位对比如下维度：

| 维度 | 说明 |
|------|------|
| 用户群体 | 直接使用该仓库的用户类型 |
| 核心价值 | 仓库对用户提供的不可替代价值 |
| 改进紧迫度 | 当前痛点对用户留存的威胁程度 |
| 实施成本 | 改进所需资源投入 |
| 价值可见性 | 用户能否直接感知到改进效果 |
| 主要受益方 | 改进主要服务的对象 |

### 评估原则

1. **用户视角优先**：只评估改进对终端用户的价值，不关注技术实现差异
2. **投入产出比量化**：每项改进标注实施成本与预期收益
3. **可落地性评估**：区分当天可上线 vs 需要较长周期
4. **价值可见性**：区分直接用户感知 vs 间接基础设施提升

### 用户价值分类

| 等级 | 含义 | 行动 |
|------|------|------|
| 🔴 高价值 | 直接影响核心用户体验，改进后用户可感知显著提升 | P0-P1 优先落地 |
| 🟡 中价值 | 改善社区治理或间接体验，用户感知较弱 | P1-P2 安排 |
| 🟢 低价值 | 长期架构演进或低频场景，用户感知极少 | P3 或 backlog |

## 二、用户痛点严重度分级

| 严重度 | 定义 | 示例 |
|--------|------|------|
| 🔴 高 | 直接影响用户能否完成任务，或构成安全隐患 | 核心功能缺失、安全漏洞 |
| 🟡 中 | 影响用户体验但不会直接导致任务失败 | 体验碎片化、反馈不清晰 |
| 🟢 低 | 非常见场景或对现有用户影响有限 | 边缘场景、新用户门槛 |

## 三、改进项价值评估模板

| 改进项 | 用户痛点 | 落地收益 | 实施成本 | 优先级 |
|--------|---------|---------|---------|--------|
| (具体改进名称) | (用户遇到的具体问题) | (改进后可量化的收益描述) | (极低/低/中/高) | P0-P3 |

### 优先级定义

| 优先级 | 时间窗口 | 特征 |
|--------|---------|------|
| **P0** | 当天可上线 / 1-2 周 | 极低成本 + 高用户价值，无阻塞依赖 |
| **P1** | 本季度内 | 中等成本，用户价值明确，但需工程化 |
| **P2** | 持续迭代 | 长期优化方向，需协调多个模块 |
| **P3** | 长期架构 | 需架构演进或生态建设，当前阶段不阻塞 |

## 四、通用改进策略缓存

### 投入产出比最高的改进模式

| 改进类型 | 典型成本 | 典型收益 | 适用场景 |
|---------|---------|---------|---------|
| **正则/规则升级** | 极低（几行代码） | 直接提升准确率 | 校验逻辑、解析逻辑 |
| **错误信息优化** | 极低 | 用户自行修正，减少沟通成本 | 任何用户可见的错误提示 |
| **查询预处理/规范化** | 低（别名映射表） | 减少无效查询，提升命中率 | 有用户输入检索的模块 |
| **文档补全** | 中 | 覆盖核心需求场景 | 知识库类项目 |
| **入口分流** | 低 | 新用户不至于被复杂规则拦截 | 有多级用户类型的项目 |

### 风险类型与缓解

| 风险 | 缓解措施 |
|------|---------|
| 规则改动引入回归 | 充分测试 + 回滚预案 |
| 误报影响正常用户 | 白名单机制 + 人工复核通道 |
| 文档补全成本超预期 | 优先补高价值内容，低频场景可长期积累 |

## 五、北极星指标推荐

### 使用侧
- 首次操作成功率
- 关键路径完成率
- 失败后的流失率

### 贡献侧
- 首次贡献完成率
- 贡献平均耗时
- 低质量 → 高质量的转化率

### 传播侧
- 内容被分享/引用次数
- 外部引用数
- 由失败转为贡献的比例

## 六、分析框架适用说明

本框架适用于评估任何开源项目或知识库系统的用户价值改进优先级。使用时应注意：

1. 数据来源应尽量覆盖真实用户场景（如实际 issues、搜索日志、用户反馈）
2. 改进项的优先级判断应基于当前阶段而非理想终局
3. "实施成本"应包含验证成本，不只是开发成本
4. 定期重新评估优先级（每次架构或生态变化后）
